# Implementation Plan

Broken into self-contained phases. Each phase should be shippable — the site works at the end of every phase, just with less in it.

---

## Phase 1 — Core renderer and layout ✓

**Goal:** A working 64×64 pixel grid on screen. Nothing moves yet, but the foundation everything else builds on is solid.

- Replace the placeholder `index.html` body with the single-page app shell
- Render the 64×64 grid using a `<canvas>` element
  - Canvas sized to fill the viewport, grid centred and scaled to fit with square pixels
  - Pixel state stored as a flat `Uint8ClampedArray` (RGBA) or a simple 2D array of colour values
  - `render()` function that writes pixel state to the canvas — called on every animation frame
- Implement the `requestAnimationFrame` loop
- Basic CSS: black background, canvas centred, no scrollbars, full viewport height
- Expose a minimal display API that all future modes will use:
  - `display.setPixel(x, y, r, g, b)`
  - `display.clear(r, g, b)`
  - `display.getPixel(x, y)`

**Done when:** A static pattern (e.g. a checkerboard or colour gradient) is visible and fills the viewport correctly on desktop and mobile.

---

## Phase 2 — UI overlay ✓

**Goal:** The mode-switching controls exist and behave correctly — visible when needed, invisible otherwise.

- Add an overlay layer (HTML elements positioned over the canvas with `pointer-events` management)
- Fade in on mouse movement or touch **outside** the grid; fade out after idle timeout (~3 seconds)
- Overlay contains:
  - Mode selector (initially just a list: Animations, Clock, Ripple, Album Art, Games)
  - Currently selected mode label
- No mode switching logic yet — just the UI shell and the fade behaviour
- CSS transitions for opacity; overlay uses `pointer-events: none` when hidden

**Done when:** Moving the mouse near the edge of the page reveals a minimal control strip that fades away on its own.

---

## Phase 3 — Animation engine + first animations ✓

**Goal:** The display comes alive. The default state is a looping animation; users can cycle through a small pool.

- Implement an animation runner: each animation is an object with `init()`, `tick(dt)`, and optionally `destroy()`
- Wire the mode selector to start/stop animations
- Implement 3–4 starter animations:
  - **Colour wash** — full-grid hue rotation, slow and hypnotic
  - **Starfield** — scrolling white dots on black, depth-sorted
  - **Fire** — bottom-up heat simulation with a warm palette
  - **Rain** — vertical green streaks on black (retro terminal feel)
- Auto-cycle through animations on a timer (e.g. every 60 seconds) unless the user has pinned one
- Add previous/next controls to the overlay

**Done when:** The display runs animations out of the box and the user can browse between them.

---

## Phase 4 — Clock modes

**Goal:** Two clock modes: digital and analogue.

### Digital clock
- Define a pixel font for digits 0–9 (5×7 or similar bitmap)
- Render HH:MM:SS centred on the grid, updating each second
- Optional: a thin progress bar along the bottom edge for sub-minute seconds

### Analogue clock
- Draw a circular face: tick marks at 12 positions
- Calculate and draw hour, minute, and second hands as pixel lines
- Smooth second hand (sub-second interpolation using `performance.now()`)
- Colour scheme that feels retro but readable

**Done when:** Both clock modes show the correct time and update live.

---

## Phase 5 — Ripple pool

**Goal:** The grid simulates a water surface; interaction creates ripples.

- Implement a 2D wave simulation on the pixel grid:
  - Two height-map buffers (current and previous frame)
  - Per-frame propagation: each cell averages its neighbours minus the previous value, times a damping factor
- Map height values to a colour palette (deep blue to white-crest)
- Mouse click and touch events inject energy at the interaction point
- Edge reflection (waves bounce off the boundary)
- Optional: gentle ambient perturbation so the pool is never completely still

**Done when:** Clicking or tapping the display creates satisfying, decaying ripples.

---

## Phase 6 — Album art (built-in covers)

**Goal:** A mode that shows pixel-rendered album covers from a built-in library.

- Define a data format for stored covers: a 64×64 array of RGB values (can be stored as a compact JS array or a small PNG decoded at runtime)
- Implement **Screamadelica** (Primal Scream, 1991) as the first cover — hand-crafted or generated by downsampling a reference image offline and encoding the result
- Add 2–3 more covers from the same era/aesthetic
- Cover select UI in the overlay: thumbnail strip or named list
- Smooth crossfade between covers

**Done when:** The album art mode displays Screamadelica and the user can switch between covers.

---

## Phase 7 — Games

**Goal:** At least one playable game on the pixel canvas.

Start with **Snake** as it maps cleanly to a grid:
- Arrow keys / WASD / swipe for direction
- Food spawns at a random empty cell
- Snake grows on collection, dies on wall or self-collision
- Score displayed using the pixel font from Phase 4
- Game over screen with restart option

Add further games (Pong, Breakout, Tetris-like) as separate sub-tasks once Snake is done.

Each game is a self-contained module following the same `init / tick / destroy` contract as animations.

**Done when:** Snake is playable start to finish.

---

## Phase 8 — Last.fm live album art

**Goal:** Enter a Last.fm username; the display updates in real time to show what's playing.

- Settings panel in the overlay: text input for Last.fm username + API key
- Store both in `localStorage`
- Poll `user.getRecentTracks` (Last.fm API) every 10–15 seconds
- If a `nowplaying` track is present, fetch its album art URL
- Downsample the image to 64×64 using an off-screen `<canvas>` with `drawImage`
- Crossfade to the new art when the track changes
- Fallback pattern (e.g. a placeholder animation) if no art is available or no track is playing

**Done when:** With valid credentials entered, the display reflects what the user is listening to.

---

## Notes

- Phases 1–3 are the critical path — everything else depends on a solid renderer and animation loop.
- Phases 4–8 are largely independent once Phase 3 is done and can be tackled in any order.
- Each phase should be committed as a logical unit with a clear before/after.
- The project structure will grow as modules are added; keep each mode in its own JS file under `assets/js/modes/`.
